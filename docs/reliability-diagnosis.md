# ShreyWS Reliability Diagnosis - 2026-07-13

## Summary

ShreyWS was investigated after several short boots and unexpected outages during the week of 2026-07-07.

The strongest finding is that the most recent previous boot did not end from a kernel panic, OOM, thermal shutdown, disk I/O failure, watchdog reset, or silent power loss. It ended through a clean systemd poweroff after the physical power key was pressed.

A separate reliability issue was confirmed: `dhcpcd` and Tailscale were fighting over `/etc/resolv.conf`. `dhcpcd` was repeatedly replacing Tailscale's resolver file with DHCP-provided DNS servers, and Tailscale was repeatedly restoring its own file for MagicDNS. This caused repeated Tailscale DNS health warnings.

Hardware checks found normal temperatures and no active filesystem failure evidence. The 500 GB backup disk `/dev/sdb`, mounted at `/srv/shreyws/backups`, passes SMART overall health but has historical bad-sector indicators and should be treated as a replacement risk.

## Symptoms

- Current boot began on 2026-07-11 at 07:49 CEST.
- Previous boots in `journalctl --list-boots` were short and frequent.
- Tailscale logged repeated DNS warnings:
  - `resolv.conf changed from what we expected`
  - `System DNS config not ideal. /etc/resolv.conf overwritten`
- `/etc/resolv.conf` alternated between Tailscale-generated content and `dhcpcd`-generated content.

### Crash Class Review

- Clean shutdown: confirmed for the previous boot.
- Power loss: not indicated for the previous boot; journal reached clean poweroff target.
- Kernel panic: no matching evidence found in previous/current kernel logs.
- OOM: no OOM kill evidence found. Some `OOM killer disabled/enabled` messages occurred around suspend/resume, which is normal kernel behavior during suspend.
- Thermal shutdown: no evidence found; current CPU and GPU temperatures are normal.
- Watchdog reset: no reset evidence found. Shutdown logs mention the hardware watchdog during systemd shutdown, which is not by itself a watchdog reset.
- Filesystem or disk failure: no fatal ext4, ATA, or I/O error evidence found in inspected logs.
- Hardware error: no MCE/EDAC fatal hardware error evidence found.

- Tailscale DNS and MagicDNS were enabled.

## Changes Made

### Installed SMART Monitoring

`smartmontools` was installed because `smartctl` was missing and disk health inspection was required.

This also enabled and started `smartmontools.service`. The daemon is active and monitoring three SMART-capable disks.

Current limitation: the default smartd mail notification path expects `/usr/bin/mail`, which is not installed. SMART warnings are visible in journald, but email delivery is not currently configured.

### Stopped dhcpcd from Rewriting resolv.conf

Backed up `/etc/dhcpcd.conf`:

```text
/etc/dhcpcd.conf.bak.20260713-145651
```

Added:

```text
# ShreyWS: Tailscale owns /etc/resolv.conf for MagicDNS; keep dhcpcd from overwriting it.
nohook resolv.conf
```

Reloaded only `dhcpcd` for `eno1`:

```bash
dhcpcd -n eno1
```

This preserves DHCP on the wired interface while preventing `dhcpcd` from owning resolver configuration. Tailscale remains the DNS owner so MagicDNS continues to work.

No reboot was performed.

If Tailscale DNS needs to be re-applied after rollback/testing:

```bash

```

Remove SMART monitoring if required:

```bash
sudo systemctl disable --now smartmontools
sudo apt-get remove smartmontools
```

No reboot is required for the rollback commands above.

## Unresolved Risks

- The 500 GB backup disk `/dev/sdb` has 152 reallocated sectors and 8 offline uncorrectable sectors. Even though SMART overall health is still `PASSED`, this disk is not a good long-term backup target.
- `smartd` cannot currently send email because `/usr/bin/mail` is missing. Alerts are visible in journald only.
- The physical reason for the power key event is not proven from logs. It may have been a person pressing the button, a case/front-panel issue, ACPI event source, or attached hardware sending a power key event.
- Earlier short boots are consistent with clean reboots/suspends, but physical context for why they were requested is outside the journal evidence.

## Recommended Next Steps

1. Replace `/dev/sdb` or move Borg backups to a healthier disk. A backup disk with offline uncorrectable sectors is not a trustworthy recovery target.
2. Configure SMART alert delivery through the existing monitoring stack or a minimal local mail path.
3. Consider changing the server's power-button behavior from `poweroff` to `ignore` or `lock` only if accidental power-button events recur. This was not changed automatically because it affects physical emergency operation.
4. Continue watching Tailscale logs for DNS warnings after DHCP renewals and reboots.
5. Add explicit disk SMART metrics to Prometheus in a later monitoring change if disk health alerting should appear in Grafana.
